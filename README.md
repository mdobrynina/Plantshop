# MOH Plant Shop — Интернет-магазин комнатных растений

## Техническое задание

**Цель:** разработать полнофункциональное веб-приложение интернет-магазина комнатных растений с ролевой моделью доступа, обеспечивающее полный цикл покупки — от просмотра каталога до оформления заказа и его обработки персоналом.

**Роли пользователей:** покупатель (CLIENT), флорист (FLORIST), администратор (ADMIN).

**Требования к системе:**
- Клиент-серверная архитектура: React SPA + Spring Boot REST API + PostgreSQL
- JWT-аутентификация со сроком действия 24 часа; пароли — BCrypt
- Адаптивный интерфейс от 360 px (Chrome, Firefox, Edge)
- Асинхронная отправка HTML-чека на e-mail через SMTP
- Поиск с живыми подсказками, фильтрация и сортировка каталога
- Корзина и избранное в гостевом режиме с последующим слиянием при входе
- Управление статусами заказов и складскими остатками на стороне флориста
- Полный CRUD каталога с загрузкой фотографий на стороне администратора

---

## 5 ключевых реализованных функций

> Это основные функции, которые демонстрируются при защите и проверяются преподавателем.

### 1. JWT-аутентификация с ролевой моделью
Три роли: `CLIENT`, `FLORIST`, `ADMIN`. Вход через `POST /api/auth/login`, токен хранится в `localStorage`, каждый запрос проходит через `JwtAuthFilter`. Защищённые маршруты на фронтенде автоматически перенаправляют на логин при 401.

**Тест:** войти как `admin@moh.ru` / `admin123` → попасть в панель администратора; войти как `florist@moh.ru` / `florist123` → попасть в рабочий стол флориста.

---

### 2. Каталог с поиском и фильтрацией
Сайдбар с категориями, поиск с debounce 250 мс и выпадающими подсказками (`GET /api/products?q=...`), фильтры по сложности ухода и цене, сортировка. Корзина и избранное работают без регистрации.

**Тест:** ввести «мон» в поиск → увидеть подсказки с монстерами; выбрать категорию «Антуриумы» в сайдбаре → отфильтрованный список.

---

### 3. Полный цикл оформления заказа
Корзина с чекбоксами и выбором количества → промокоды (`MOH10`, `GREEN20`, `PLANTS15`) → оформление с валидацией полей, автозаполнением адреса из предыдущего заказа → `POST /api/orders` → списание остатков → HTML-чек на e-mail → редирект в профиль через 3 секунды.

**Тест:** добавить товар в корзину, применить промокод `GREEN20`, оформить заказ → проверить e-mail и вкладку «Мои заказы» в профиле.

---

### 4. Рабочий стол флориста
Список входящих заказов с фильтром по типу доставки. Управление статусами жизненного цикла: **Новый → В обработке → Готов → Доставлен**. Складской учёт: при оформлении заказа остаток автоматически уменьшается; флорист видит актуальные остатки и может добавлять новые позиции.

**Тест:** войти как флорист → найти новый заказ → последовательно перевести по статусам → убедиться, что покупатель видит обновлённый статус в профиле.

---

### 5. Панель администратора
Полный CRUD каталога товаров: добавление, редактирование, удаление, загрузка фотографий (`multipart/form-data`). Просмотр всех пользователей и создание учётных записей сотрудников. Сводная аналитика по заказам.

**Тест:** войти как администратор → добавить новый товар с фото → убедиться, что он появился в каталоге с правильными данными и изображением.

# Пояснительная записка к проекту

## Введение

Рынок комнатных растений активно переходит в онлайн-формат: покупатели ценят возможность выбирать растения не выходя из дома, получать советы по уходу и отслеживать заказы. Вместе с тем большинство существующих решений либо представляют собой универсальные маркетплейсы без специализации, либо не предоставляют покупателю достаточной информации об уходе за конкретным растением. Небольшие цветочные магазины зачастую не имеют собственной цифровой инфраструктуры и ведут учёт вручную.

В связи с этим была сформулирована цель курсового проекта:

> Разработать веб-приложение интернет-магазина комнатных растений с ролевой моделью доступа, обеспечивающее полный цикл покупки — от просмотра каталога до оформления заказа и его обработки персоналом.

Для достижения поставленной цели необходимо решить следующие задачи:

1. Определить сценарии использования системы для трёх типов пользователей: покупателя, флориста и администратора;
2. Спроектировать архитектуру приложения, модель данных и пользовательский интерфейс;
3. Реализовать и протестировать прототип полнофункционального веб-приложения с клиентской и серверной частями.

---

## 1. Функциональное назначение программного средства

### 1.1 Классы решаемых задач

Разрабатываемое программное средство решает следующие классы задач:

- **Задачи поиска и навигации** — поиск товаров по названию с живыми подсказками, фильтрация каталога по категории, сложности ухода и цене, сортировка;
- **Задачи управления данными** — ведение корзины и избранного (в том числе для неавторизованных пользователей с последующим слиянием), оформление и хранение заказов;
- **Задачи анализа и контроля** — отслеживание складских остатков, управление статусами заказов флористом, администрирование товарного каталога;
- **Задачи оптимизации трудозатрат** — автоматическая отправка чека на e-mail, автозаполнение данных доставки из предыдущего заказа, быстрый вход для сотрудников.

### 1.2 Функциональное назначение

Программа предназначена для автоматизации процессов онлайн-торговли комнатными растениями и обеспечения взаимодействия между покупателями и персоналом магазина.

**Покупатель** имеет возможность:

- просматривать каталог с фильтрацией и поиском;
- добавлять товары в корзину и избранное (в том числе без регистрации);
- применять промокоды со скидкой;
- оформлять заказы с выбором способа доставки и оплаты;
- получать чек на электронную почту после оформления заказа;
- отслеживать статус заказов и просматривать советы по уходу за купленными растениями в личном кабинете.

**Флорист** имеет возможность:

- просматривать входящие заказы с фильтрацией по типу доставки;
- переводить заказы по статусам жизненного цикла (Новый → В обработке → Готов → Доставлен);
- управлять складскими остатками товаров, добавлять новые позиции.

**Администратор** имеет возможность:

- управлять полным каталогом товаров (добавление, удаление, редактирование, загрузка фотографий);
- просматривать сводную аналитику по заказам и пользователям;
- создавать учётные записи сотрудников.

### 1.3 Нефункциональные требования, ограничения и интерфейсы

- Взаимодействие клиентской и серверной частей осуществляется через **REST API** (JSON, HTTP/HTTPS);
- Аутентификация реализована на основе **JWT-токенов** со сроком действия 24 часа;
- Интерфейс должен корректно отображаться в современных браузерах (Chrome, Firefox, Edge) на экранах шириной от 360 px;
- Серверная часть должна стартовать не более чем за 10 секунд на локальной машине разработчика;
- Пароли пользователей хранятся в виде **BCrypt**-хешей; чувствительные параметры конфигурации (ключи JWT, SMTP-пароль) вынесены в `application.properties` и не включаются в систему контроля версий;
- Отправка e-mail выполняется **асинхронно**, чтобы не блокировать ответ сервера при оформлении заказа.

## 2. Проектная часть

### 2.1 Архитектура приложения

Для обеспечения независимости клиентской части от серверной реализации и удобства раздельного развёртывания была выбрана **многоуровневая клиент-серверная архитектура** с разделением на три слоя:

```
┌─────────────────────────────────┐
│   Frontend  (React + Vite)      │  порт 5173 (dev) / статика (prod)
│   SPA, CSS-переменные           │
└────────────────┬────────────────┘
                 │ REST API (JSON)
┌────────────────▼────────────────┐
│   Backend  (Spring Boot 3)      │  порт 8080
│   Controller → Service → Repo  │
└────────────────┬────────────────┘
                 │ JPA / Hibernate
┌────────────────▼────────────────┐
│   База данных  (PostgreSQL 18)  │
└─────────────────────────────────┘
```

Основными компонентами серверной части являются:

- **Controllers** — обработка HTTP-запросов, валидация входных данных (`AuthController`, `ProductController`, `CartController`, `OrderController`, `AdminController`, `FloristController`, `FavoriteController`);
- **Services** — бизнес-логика (`OrderService` с проверкой остатков, `EmailService` с асинхронной отправкой чеков);
- **Repositories** — доступ к данным через Spring Data JPA;
- **Security** — фильтр JWT (`JwtAuthFilter`), конфигурация Spring Security с ролевой моделью (ADMIN / FLORIST / CLIENT).

Основными компонентами клиентской части являются:

- **Pages** — страницы приложения (каталог, корзина, оформление, профиль, панели администратора и флориста);
- **Hooks** — `useCart`, `useFavorites` с логикой слияния гостевого состояния при входе;
- **API-слой** — централизованный модуль `api.js` с перехватчиком JWT и авто-разлогином при 401.

### 2.2 Проектирование пользовательского интерфейса

В соответствии с требованиями интерфейс реализован в виде **одностраничного приложения (SPA)** с навигацией через React Router. Дизайн построен на системе CSS-переменных (цвета, типографика, отступы), что обеспечивает визуальную согласованность всех страниц.

Основные экраны:

| Экран | Описание |
|---|---|
| Главная | Hero-блок, популярные категории, преимущества, о нас, доставка, отзывы |
| Каталог | Сайдбар с категориями, поиск с живыми подсказками, фильтры, карточки товаров |
| Страница товара | Описание, характеристики ухода, добавление в корзину и избранное |
| Корзина | Список позиций, выбор товаров, промокод, итоговая сумма |
| Оформление заказа | Контактные данные, доставка, оплата картой с маскировкой полей |
| Личный кабинет | Вкладки: мои заказы, мои растения (советы по уходу), личные данные |
| Панель флориста | Список заказов с управлением статусами, склад с фильтрацией |
| Панель администратора | Управление каталогом, загрузка фото, просмотр пользователей |

### 2.3 Модель данных

Основными сущностями системы являются:

| Сущность | Ключевые атрибуты |
|---|---|
| `User` | id, email, password (BCrypt), firstName, lastName, role, birthDate |
| `Product` | id, name, category, categoryName, price, stock, inStock, image, care, light, watering, height, description |
| `Order` | id, user, status, deliveryType, address, phone, total, createdAt |
| `OrderItem` | id, order, product, quantity, price |
| `Favorite` | id, user, product |
| `CartItem` | id, user, product, quantity |

Связи между сущностями:

- `User` 1 → N `Order`
- `Order` 1 → N `OrderItem`
- `OrderItem` N → 1 `Product`
- `User` 1 → N `Favorite`
- `User` 1 → N `CartItem`

### 2.4 Моделирование работы приложения

**Сценарий «Оформление заказа»:**

```
Покупатель  →  CartPage         : выбрать товары, применить промокод
Покупатель  →  CheckoutPage     : заполнить данные доставки и оплаты
CheckoutPage → OrderController  : POST /api/orders
OrderController → OrderService  : createOrder()
OrderService → ProductRepository: проверить и уменьшить складской остаток
OrderService → OrderRepository  : сохранить заказ
OrderService → EmailService     : sendOrderReceipt() [async]
EmailService → Yandex SMTP      : отправить HTML-письмо с чеком
OrderController → CheckoutPage  : { id, status }
CheckoutPage → ProfilePage      : редирект на вкладку «Мои заказы» через 3 с
```

**Сценарий «Вход с сохранением гостевой корзины»:**

```
Header      →  LoginPage        : ввести email / password
LoginPage   →  AuthController   : POST /api/auth/login
AuthController → UserDetailsService : проверить BCrypt-хеш
AuthController → JwtUtil        : сгенерировать токен
LoginPage   →  localStorage     : сохранить user + token
useCart(user)                   : обнаружить смену null → user
useCart     →  CartController   : GET /api/cart (корзина на сервере)
useCart     →  CartController   : POST /api/cart/items (гостевые позиции)
useCart     →  CartController   : GET /api/cart (объединённая корзина)
```

В качестве среды разработки была выбрана **IntelliJ IDEA 2024**. При разработке использованы следующие языки и технологии:

**Backend:**

- Java 21

- JavaScript (ES2022+)
- React 18 + React Router 6
- Vite 5
- CSS (переменные, Grid, Flexbox — без сторонних UI-библиотек)
